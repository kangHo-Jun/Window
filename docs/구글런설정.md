## 3. Secret Manager 설정

### 등록 대상

- Secret 이름: `GOOGLE_PRIVATE_KEY`

### 등록 명령

`.env.local`에 `GOOGLE_PRIVATE_KEY`가 이미 들어있다면 아래처럼 등록합니다.

```bash
cd '/Users/zart/Library/Mobile Documents/com~apple~CloudDocs/프로젝트/창호/window-estimate-system'

set -a
source .env.local
set +a

printf '%s' "$GOOGLE_PRIVATE_KEY" | gcloud secrets create GOOGLE_PRIVATE_KEY --data-file=-
```

이미 Secret이 있으면 새 버전만 추가합니다.

```bash
printf '%s' "$GOOGLE_PRIVATE_KEY" | gcloud secrets versions add GOOGLE_PRIVATE_KEY --data-file=-
```

### Cloud Run 서비스 계정 권한

Cloud Run 기본 서비스 계정이 Secret을 읽을 수 있어야 합니다.

```bash
gcloud secrets add-iam-policy-binding GOOGLE_PRIVATE_KEY \
  --member="serviceAccount:747058361639-compute@developer.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor"
```

## 4. GitHub 연동 (Cloud Build 트리거)

### 대상 저장소

- 저장소: `kangHo-Jun/Window`
- 브랜치 패턴: `^main$`
- 구성 파일: `window-estimate-system/cloudbuild.yaml`

### 현재 상태

- `cloudbuild.yaml` 생성 완료
- `default_logs_bucket_behavior: REGIONAL_USER_OWNED_BUCKET` 설정 완료
- GitHub 저장소 원격 확인 완료
- Cloud Build 트리거 자동 생성은 아직 미완료

### 미완료 사유

현재 `Window` 저장소가 Cloud Build에 repository mapping으로 연결되지 않아 아래 오류가 발생했습니다.

```text
FAILED_PRECONDITION: Repository mapping does not exist.
```

### 먼저 해야 할 작업

아래 콘솔에서 GitHub 저장소 `kangHo-Jun/Window`를 Cloud Build에 연결해야 합니다.

```text
https://console.cloud.google.com/cloud-build/triggers;region=global/connect?project=747058361639
```

### 저장소 연결 후 트리거 생성 명령

```bash
cd '/Users/zart/Library/Mobile Documents/com~apple~CloudDocs/프로젝트/창호/window-estimate-system'

gcloud builds triggers create github \
  --name=window-estimate-main \
  --description='Build and deploy window-estimate to Cloud Run on push to ^main$' \
  --repo-name=Window \
  --repo-owner=kangHo-Jun \
  --branch-pattern='^main$' \
  --build-config=cloudbuild.yaml \
  --include-logs-with-status \
  --service-account='projects/stacking-492708/serviceAccounts/747058361639-compute@developer.gserviceaccount.com' \
  --project=stacking-492708
```

### cloudbuild.yaml 현재 내용

```yaml
options:
  default_logs_bucket_behavior: REGIONAL_USER_OWNED_BUCKET

steps:
  - name: 'gcr.io/cloud-builders/gcloud'
    args:
      - 'run'
      - 'deploy'
      - 'window-estimate'
      - '--source'
      - '.'
      - '--region'
      - 'asia-northeast3'
      - '--platform'
      - 'managed'
      - '--allow-unauthenticated'
      - '--project'
      - 'stacking-492708'
```

## 6. 자주 발생하는 오류 및 해결법

### 1) Secret 접근 권한 오류

오류 예시:

```text
Permission denied on secret ... The service account used must be granted the 'Secret Manager Secret Accessor' role
```

해결:

```bash
gcloud secrets add-iam-policy-binding GOOGLE_PRIVATE_KEY \
  --member="serviceAccount:747058361639-compute@developer.gserviceaccount.com" \
  --role="roles/secretmanager.secretAccessor"
```

### 2) Cloud Build 트리거 생성 실패

오류 예시:

```text
FAILED_PRECONDITION: Repository mapping does not exist.
```

해결:

- Cloud Build 콘솔에서 GitHub 저장소를 먼저 연결
- 연결 후 `gcloud builds triggers create github ...` 재실행

### 3) 로컬 `npm run build` 시 Turbopack 포트 바인딩 오류

오류 예시:

```text
Operation not permitted (os error 1)
```

해결:

- 로컬 샌드박스 환경에서 발생할 수 있음
- 실제 검증은 권한 확장 환경 또는 Cloud Run 원격 빌드 기준으로 확인

### 4) Google Fonts/폰트 로더 관련 빌드 오류

해결:

- `next/font/google` 대신 로컬 폰트 또는 `@font-face` 사용
- 현재 프로젝트는 `globals.css`에서 로컬 폰트 방식 사용

### 5) `NEXT_PUBLIC_*` 값이 운영에서 다르게 보이는 문제

원인:

- `NEXT_PUBLIC_*` 변수는 빌드 시점에 주입됨

해결:

- Cloud Run 빌드 시점에 값이 들어가도록 구성
- 또는 런타임 의존값이면 서버 환경변수 방식으로 구조 변경
